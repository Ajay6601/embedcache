Examples

Everything below is copy-paste ready. The pattern is always the same: run embedcache serve -upstream <your backend>, then point your client's base_url at the proxy.

Ollama (verified live)

embedcache serve -upstream http://localhost:11434

curl localhost:8090/v1/embeddings -H "Content-Type: application/json" \
  -d '{"model": "all-minilm", "input": "hello world"}'

All livecheck assertions pass against Ollama 0.22 / all-minilm, including base64 encoding and mixed-batch mapping.

vLLM

# vLLM serving an embedding model on :8000
vllm serve intfloat/e5-mistral-7b-instruct

embedcache serve -upstream http://localhost:8000

vLLM's OpenAI-compatible server exposes /v1/embeddings; embedcache mirrors whatever path your client calls, so no path configuration is needed.

Text Embeddings Inference (TEI)

docker run -p 8080:80 ghcr.io/huggingface/text-embeddings-inference:latest \
  --model-id BAAI/bge-large-en-v1.5

embedcache serve -upstream http://localhost:8080
TEI note: use its OpenAI-compatible /v1/embeddings route (available in current TEI). If you use -auth-mode verify, confirm your TEI version serves /v1/models; otherwise use allowlist mode.

OpenAI

# forward each caller's own API key (default)
embedcache serve -upstream https://api.openai.com

# or pin one org-wide key and validate callers yourself
embedcache serve -upstream https://api.openai.com \
  -upstream-api-key $OPENAI_API_KEY \
  -auth-mode allowlist -api-keys-file team-keys.txt

Google Gemini (verified live)

embedcache serve \
  -upstream https://generativelanguage.googleapis.com/v1beta/openai \
  -upstream-api-key $GEMINI_API_KEY

Set your client's base_url to http://localhost:8090 (without /v1, since Google's compat endpoint uses bare /embeddings, and embedcache mirrors the client path). Current embedding model: gemini-embedding-001 (3072 dims, so at ~67 KB per vector, every cache hit also skips a full internet round-trip).

Voyage AI: the Anthropic/Claude stack

Anthropic offers no embeddings API of its own and officially recommends Voyage AI. Voyage's /v1/embeddings is wire-compatible, and its extra parameters are handled correctly:

embedcache serve -upstream https://api.voyageai.com \
  -upstream-api-key $VOYAGE_API_KEY

curl localhost:8090/v1/embeddings -H "Content-Type: application/json" \
  -d '{"model": "voyage-4", "input": "hello", "input_type": "query"}'
Voyage returns different vectors for the same text under input_type: "query" vs "document". embedcache forwards provider-specific parameters verbatim and includes them in the cache identity, so query-typed and document-typed embeddings never collide. Same handling applies to output_dtype, output_dimension, and any other extra field.

Azure OpenAI

Azure routes by deployment path and version query string, and authenticates with an api-key header, and embedcache mirrors all three:

embedcache serve -upstream https://YOUR-RESOURCE.openai.azure.com

# client base_url points at the proxy with the full deployment path
curl "localhost:8090/openai/deployments/YOUR-DEPLOYMENT/embeddings?api-version=2024-02-01" \
  -H "api-key: $AZURE_OPENAI_KEY" -H "Content-Type: application/json" \
  -d '{"input": "hello", "model": "text-embedding-3-small"}'
The query string and api-key header are forwarded upstream; the api-key value is also used as the caller identity for allowlist auth and per-caller accounting.

curl

# watch the cache work
curl -s -D - -o /dev/null localhost:8090/v1/embeddings \
  -H "Content-Type: application/json" \
  -d '{"model":"all-minilm","input":"hello"}' | grep -i x-embedcache
# X-Embedcache-Status: miss   (first time)
# X-Embedcache-Status: hit    (every time after)

Python (OpenAI SDK)

from openai import OpenAI

client = OpenAI(base_url="http://localhost:8090/v1", api_key="sk-...")
resp = client.embeddings.create(
    model="text-embedding-3-small",
    input=["first text", "second text"],
)
# resp.usage.prompt_tokens == only what THIS call was billed upstream

Docker Compose (embedcache + Ollama)

services:
  ollama:
    image: ollama/ollama
    volumes: ["ollama:/root/.ollama"]

  embedcache:
    build: https://github.com/Ajay6601/embedcache.git
    command: >
      serve -listen :8090 -upstream http://ollama:11434
      -persist /data/cache.snap -request-log /data/requests.jsonl
      -admin-token ${EC_ADMIN_TOKEN}
    ports: ["8090:8090"]
    volumes: ["ecdata:/data"]
    depends_on: [ollama]

volumes: { ollama: {}, ecdata: {} }

Kubernetes

apiVersion: apps/v1
kind: Deployment
metadata: { name: embedcache }
spec:
  replicas: 1                     # cache is per-pod; see docs § limitations
  selector: { matchLabels: { app: embedcache } }
  template:
    metadata: { labels: { app: embedcache } }
    spec:
      containers:
      - name: embedcache
        image: your-registry/embedcache:latest
        args: ["serve", "-listen", ":8090",
               "-upstream", "http://vllm.inference.svc:8000",
               "-persist", "/data/cache.snap",
               "-admin-token", "$(ADMIN_TOKEN)"]
        env:
        - name: ADMIN_TOKEN
          valueFrom: { secretKeyRef: { name: embedcache, key: admin-token } }
        ports: [{ containerPort: 8090 }]
        livenessProbe: { httpGet: { path: /healthz, port: 8090 } }
        volumeMounts: [{ name: data, mountPath: /data }]
      volumes:
      - name: data
        persistentVolumeClaim: { claimName: embedcache-data }
---

systemd

[Unit]
Description=embedcache embedding proxy
After=network-online.target

[Service]
EnvironmentFile=/etc/embedcache/env    # contains ADMIN_TOKEN=...
ExecStart=/usr/local/bin/embedcache serve \
  -upstream http://localhost:8000 \
  -persist /var/lib/embedcache/cache.snap \
  -admin-token ${ADMIN_TOKEN}
DynamicUser=yes
StateDirectory=embedcache
Restart=on-failure

[Install]
WantedBy=multi-user.target

Workflow: audit your existing logs (no install in the request path)

# 1. dump embedding request bodies from whatever you have: gateway logs,
#    app logs, LiteLLM logs, one JSON object per line
# 2. run the analyzer
embedcache analyze -default-price 0.05 requests.jsonl

# machine-readable, for a dashboard or a slack bot
embedcache analyze -json requests.jsonl | jq '{rate: .duplicate_rate, wasted_usd: .wasted_usd_est}'

The number it prints is the exact share an exact-match cache would have absorbed; the analyzer and the live proxy agreed to the item when validated on identical traffic.

Workflow: full production config

embedcache serve \
  -listen 127.0.0.1:8090 \
  -upstream http://localhost:8000 \
  -admin-token  "$(cat /etc/embedcache/admin-token)" \
  -auth-mode    allowlist -api-keys-file /etc/embedcache/keys.txt \
  -ttl          720h \
  -persist      /var/lib/embedcache/cache.snap \
  -request-log  /var/log/embedcache/requests.jsonl \
  -pricing      /etc/embedcache/pricing.json

# pricing.json: give self-hosted models an amortized GPU cost so the
# waste report speaks in dollars:
{ "bge-large-en-v1.5": 0.03, "default": 0.05 }

Workflow: redeploying a model under the same name

# cached vectors are only valid for the weights that produced them,
# flush on deploy, or run with -ttl as a safety net
curl -X POST -H "Authorization: Bearer $ADMIN_TOKEN" localhost:8090/_ec/flush

# take a snapshot before risky changes
curl -X POST -H "Authorization: Bearer $ADMIN_TOKEN" localhost:8090/_ec/snapshot

Workflow: Prometheus + alerts

scrape_configs:
  - job_name: embedcache
    static_configs: [{ targets: ["embedcache:8090"] }]
    authorization: { credentials: "your-admin-token" }

# alert when the backend is down (cache keeps serving hits meanwhile)
- alert: EmbedcacheCircuitOpen
  expr: embedcache_breaker_open == 1
  for: 1m

Workflow: multi-tenant cost accounting

# give each team its own key
embedcache serve -upstream http://localhost:8000 \
  -auth-mode allowlist -api-keys team-search-x9f,team-rag-k2m,team-agents-p7q

# per-caller usage shows up in stats (keys appear as short hashes)
curl -s -H "Authorization: Bearer $ADMIN_TOKEN" localhost:8090/_ec/stats \
  | jq .per_caller

Every team shares one cache (team A's ingestion warms team B's queries) while usage is attributed per key.

Add hard budgets so a runaway agent can't burn the shared backend:

embedcache serve -upstream http://localhost:8000 \
  -auth-mode allowlist -api-keys-file keys.txt \
  -budgets-file budgets.json -budget-window 24h

# budgets.json
{ "team-agents-p7q": 500000, "default": 2000000 }

# when team-agents exhausts its 500k tokens: new computation gets 429,
# but everything already cached keeps serving
