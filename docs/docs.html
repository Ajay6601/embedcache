Documentation

embedcache is a caching, deduplicating, cost-accounting proxy for OpenAI-compatible embedding APIs. One static binary, pure Go stdlib, three subcommands: serve (the proxy), analyze (offline waste report), report (live waste report).

Install

From source (Go 1.22+)

go install github.com/Ajay6601/embedcache/cmd/embedcache@latest

Clone and build

git clone https://github.com/Ajay6601/embedcache
cd embedcache
go build -o embedcache ./cmd/embedcache

Docker

docker build -t embedcache https://github.com/Ajay6601/embedcache.git
docker run -p 8090:8090 embedcache serve -listen :8090 -upstream http://host.docker.internal:11434

The binary has zero runtime dependencies — copy it anywhere and run it.

Quickstart

Start the proxy in front of your embedding backend:

# self-hosted engines
embedcache serve -upstream http://localhost:11434                # Ollama
embedcache serve -upstream http://localhost:8000                 # vLLM / TEI

# hosted providers
embedcache serve -upstream https://api.openai.com                # OpenAI
embedcache serve -upstream https://generativelanguage.googleapis.com/v1beta/openai \
                 -upstream-api-key $GEMINI_API_KEY               # Gemini

Then point your client's base_url at the proxy instead of the provider. Your Authorization header is forwarded upstream (or pinned with -upstream-api-key). Any request path ending in /embeddings is cached; everything else passes through untouched, so embedcache can front a full OpenAI-compatible server.

# Python
client = OpenAI(base_url="http://localhost:8090/v1", api_key="sk-...")
client.embeddings.create(model="text-embedding-3-small", input=["hello", "world"])

See Examples for JavaScript, Go, LangChain, LlamaIndex, Docker Compose, and Kubernetes recipes.

Response headers

Every embeddings response tells you exactly what happened:

HeaderMeaning
X-Embedcache-Statushit (fully served from cache), miss (fully computed upstream), or partial (mixed batch)
X-Embedcache-Hitsitems served from cache
X-Embedcache-Missesitems computed upstream by this request
X-Embedcache-Coalesceditems served by waiting on another in-flight request's computation
X-Embedcache-Saved-Tokenstokens this request did not pay for
Billing semantics: usage.prompt_tokens in the response reflects only what this request was actually billed upstream. Cache hits and coalesced items cost zero. Your existing usage accounting keeps working — it just gets smaller.

How it works

  1. Fingerprint. Each input item is hashed with its full identity: SHA-256(model, dimensions, encoding_format, provider params, content). Provider-specific parameters (Voyage's input_type/output_dtype, Mistral extras, vLLM options) are forwarded upstream verbatim and included in the key — the same text under different params gets different vectors, so it gets different entries. Two items share a cache entry only when the upstream response is guaranteed interchangeable. Token-array inputs are supported and never collide with text.
  2. Split the batch. Cached items are answered from memory as byte-exact replays of the upstream's original JSON. Items currently being computed by any other request are coalesced — the request waits for that result instead of recomputing. Duplicates within one batch are computed once.
  3. Forward the remainder. Only never-seen items go upstream, as one batch. Results are cached, distributed to all waiters, and reassembled at their original indices.
Byte-exact means byte-exact: the cached value is the raw JSON of the upstream's embedding field, replayed verbatim. No float parsing, no re-encoding, no precision drift. encoding_format=base64 and float are cached as separate entries for the same reason.

The waste analyzer

The analyzer is how you find out whether embedcache is worth deploying — before deploying it. Feed it JSONL request logs (one embedding request per line, either raw bodies or wrapped under body/request/payload):

embedcache analyze requests.jsonl
embedcache analyze -json requests.jsonl | jq .duplicate_rate
cat logs/*.jsonl | embedcache analyze

It fingerprints every input exactly like the live proxy, so its duplicate count is exactly what the cache would have absorbed — validated to agree with a live proxy to the item on identical traffic.

# useful flags
-default-price 0.05   # $/1M tokens for self-hosted models (amortized GPU cost)
-pricing costs.json   # per-model prices
-normalize trim       # preview what normalization would add
-top 25               # show more of the top duplicated inputs

To produce logs the analyzer can read, run the proxy with -request-log, or dump requests from your own gateway/app — the format is forgiving.

Chunk-diff engine

Exact-match caching only helps if re-ingesting a document produces the same chunk text as before. Fixed-size chunking (slicing by byte offset) breaks that: any edit shifts every chunk boundary after it, so a small edit re-embeds the whole rest of the document. embedcache chunk splits text at content-defined boundaries instead — an edit only changes the one or two chunks touching it.

embedcache chunk document.md > chunks.jsonl
# one JSON line per chunk: {"hash": "...", "size": 743, "text": "..."}
cat chunks.jsonl | jq -r .text | while read -r c; do
  # feed each chunk into your batched /v1/embeddings calls
done

Flags: -min/-avg/-max bound chunk size in bytes (defaults 200/800/2000, tuned for paragraph-scale text). Reads a file argument or stdin.

Measured on one real, live Wikipedia article with one realistic single-sentence edit — CHUNKDIFF.md: content-defined chunking absorbed 93.8% of the re-ingest from cache; fixed-size chunking absorbed 28.6% on the identical edit.

Security

Two separate layers, both off by default for painless first contact, both required for untrusted callers:

1. Admin token

embedcache serve -upstream ... -admin-token "$(openssl rand -hex 24)"

Without it, anyone who can reach the port can flush your cache and read usage stats.

2. Data-plane key validation

Cache hits never touch the upstream — so without this, a revoked or missing API key can still read cached vectors. Two modes:

# allowlist: the proxy owns the key list (self-hosted backends with no auth)
embedcache serve -upstream http://localhost:8000 \
  -auth-mode allowlist -api-keys-file keys.txt

# verify: keys are checked against the upstream (hosted providers)
embedcache serve -upstream https://api.openai.com -auth-mode verify

verify probes GET /v1/models with the caller's key and caches the verdict for -auth-cache-ttl (rejections for 1 minute; fails closed if the upstream is unreachable). Keys are stored in memory only as SHA-256 hashes.

embedcache listens in plaintext by design — terminate TLS in front of it (Caddy, nginx, or your ingress). The request log stores raw input text; treat it with the same care as the documents it came from.

Resilience

Embedding calls are idempotent, so transient upstream failures (network errors, 5xx, 429) are retried with exponential backoff and jitter, honoring Retry-After. Definitive 4xx errors are never retried and pass through verbatim.

Sustained failure trips a circuit breaker: after -breaker-threshold consecutive faults, requests needing the upstream fail fast with 503 instead of stacking timeouts. After -breaker-cooldown, one probe decides whether to close it.

The property that matters: cache hits keep serving while the circuit is open. A warmed embedcache acts as a read-through fallback during backend outages — the outage degrades misses, not the service.

Persistence & TTL

embedcache serve -upstream ... -persist /var/lib/embedcache/cache.snap -ttl 720h
  • The snapshot is loaded at start, written on graceful shutdown, every -persist-interval, and on demand via POST /_ec/snapshot. Writes are atomic (temp file + rename). A hard crash loses at most the entries since the last snapshot — for a cache, that means a few extra misses, never wrong data.
  • -ttl bounds staleness. If your model files can change under the same model name (common with self-hosted fine-tune iterations), set a TTL or call /_ec/flush on deploy — otherwise entries never expire, because for a fixed model the correct answer never changes.
  • Memory: entries are bounded by -max-entries and -max-memory-mb with LRU eviction, sharded 16 ways.

Guarantees

  • A cache hit returns the byte-identical JSON value the upstream returned when the entry was created — verified by fuzzing, at 430k-request scale, and across crash-restarts.
  • Batch responses preserve input order and index mapping under any mix of hits, misses, and intra-batch duplicates.
  • A unique input in flight is computed upstream at most once across all concurrent requests.
  • Upstream errors pass through with their original status and body, and are never cached.
  • Exact-match only: the wrong-vector rate of the cache is 0% by construction.

Limitations (read this)

  • Fixed-size chunking defeats exact-match caching. Slicing a document by byte offset means any edit shifts every downstream chunk, so a naive re-chunk re-embeds everything — measured at 0% absorbed in E4. Use embedcache chunk (content-defined chunking) instead: edits only perturb the chunks they touch — 93.8% absorbed on a real edit to a real article, see CHUNKDIFF.md.
  • Semantically similar ≠ identical. "reset my password" and "how to reset password" are different entries. Opt-in normalization (trim,collapse,lowercase) widens matching slightly; semantic caching is deliberately not offered (see FAQ).
  • Single-node cache. Each instance has its own memory; there is no shared backend yet. One instance handles ~39k hit req/s, which covers a lot of ground before this matters.
  • Your savings depend on your workload. A real agent workload measured 49.7% duplicate tokens; yours may be more or less. That is exactly what embedcache analyze is for — measure before you deploy.

FAQ

Why not semantic caching?

Similarity-threshold caching returns wrong answers at some rate — published evaluations show meaningful false-hit rates even at high thresholds. For embeddings specifically, wrong vectors silently poison your vector store. Exact-match has a 0% wrong-answer rate, and the analyzer proves how much it still saves.

Is it a library or a framework?

Neither — it's a service, like Redis or nginx. Your apps in any language talk to it over HTTP without importing anything. That's the point: LangChain and LlamaIndex already dedupe inside Python; embedcache does it for your whole stack.

What happens when embedcache itself dies?

Point your clients at the upstream directly — the API is passthrough-compatible, so the failure mode is "you pay full price again," never "you're down." Restore from the snapshot when it's back.

Does it work with chat completions too?

Non-embedding routes pass through unmodified. Caching chat responses is a different problem (streaming, sampling temperature, wrong-answer risk) and out of scope on purpose.

How do I know it's not lying about savings?

Everything is measured and reproducible: the repo ships the controlled experiments (re-run by CI on every push), a 430k-request production simulation, and a real agent workload on live data with zero constructed duplicates.
